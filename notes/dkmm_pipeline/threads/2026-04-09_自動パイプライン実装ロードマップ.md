# 自動パイプライン実装ロードマップ

> dkmm.com を「上がってきたものを読んで気になったら反応するだけ」で育てるパイプラインの実装計画。
> 2026-04-09 セッションで構想固まったやつを帰りタスクとして書き残す。
> 次に PC 触れた時 or vision 着手セッションでこのファイルを起点にする。

## 全体像

```
X bookmark 追加
  ↓ Cloud Scheduler (毎日 cron)
fetch_bookmarks → Secret Manager から OAuth 読込・rotate 後書戻
  ↓ 新着検知
Claude API で thread draft 生成 (ねーさん文体)
  ↓
Slack channel に投稿 (要約 + 仮説 + 残ってる疑問)
  ↓ dkm がスマホで読む
  ↓ 気になったらスレッドで反応
Cloud Run service が Slack webhook 受信 → Claude で対話
  ↓ 対話の結論を整形
GitHub repo に commit / PR (notes/{topic}/threads/ に追記 or 新規)
```

すべて GCP 上で動く想定。dkm 側はスマホの Slack だけで完結。

## 設計上の制約・要件

- **★スマホ完結**: fetch から thread 更新まで全工程がスマホ(Slack)で完結。PC ゼロ依存
- **★主舞台 Slack**: 通知の即時性とテンポが GitHub Mobile より良い
- **★OAuth state は Secret Manager**: X refresh_token は rotate するから、env var 渡しでは不可
- **★ねーさんのトークン効率**: 「点と点を線で繋ぐ」作業のスケーラビリティを決める前提条件(後述)
- **コスト**: 個人運用なので無料枠+ほぼゼロ円
- **既存パターン流用**: 別途持っている Cloud Run + Slack bot 構成(FastAPI + slack-sdk + Dockerfile + deploy.sh)をベースに

## ★ 最重要の隠れた要件: トークン効率

vision pipeline は単に「動く」だけでは不足。**「点と点を線で繋ぐ」操作がトークン経済的に成立すること** が前提条件。

具体的には:
- 新しいブックマーク→既存スレッドとの関連発見を、毎回 full read で探したらコスト爆発
- Slack 対話のたびに過去 thread 全部 context 投入したら毎回数万トークン消費
- 「線を引く」のスケーラビリティを決めるのはここ

実装上の対応(Phase 5-6 で必須):
- `notes/_index.md` のような **軽量カタログ**(タイトル+keyword要約だけ) を維持して毎セッション最初に読む
- もしくは thread embedding を Vertex AI で生成 → Firestore / GCS に保存 → 類似度検索
- 大ファイル(`bookmarks/latest.md`, `analysis/*.md`)は Grep 先行で targeted Read
- 重い検索は subagent (Haiku) に委譲して main context を節約

## 実装フェーズ(段階的)

### Phase 1: スケルトン構築 ★最初

動く最小単位を作る。Hello world 相当でいい。

- [ ] `cloud_run/` ディレクトリ作成
- [ ] `cloud_run/main.py` — FastAPI でルート2つ:
  - `GET /health` → `{"status":"ok"}`
  - `POST /slack/events` → 即200返してログ吐くだけ
- [ ] `cloud_run/Dockerfile` — bnv パターンのコピー(python:3.11-slim, uvicorn 起動)
- [ ] `cloud_run/requirements.txt` — fastapi, uvicorn, slack-sdk
- [ ] `cloud_run/deploy.sh` — bnv のコピー、サービス名は `dkmm-bookmark-bot` 等
- [ ] gcloud auth & プロジェクト確認(既存プロジェクト流用、ID は memory 参照)
- [ ] 初回デプロイ → URL 取得 → `/health` 疎通確認

### Phase 2: Slack App 登録

- [ ] api.slack.com/apps で新規 App 作成
- [ ] OAuth scopes: `app_mentions:read`, `chat:write`
- [ ] Workspace に install して `xoxb-...` token 取得
- [ ] Signing secret 取得
- [ ] Cloud Run の env var に投入(Phase 3 で Secret Manager 化する)
- [ ] Event Subscriptions: Request URL に `https://.../slack/events`
- [ ] テストチャンネル作成 → bot を invite
- [ ] メンションでエコー返るか確認

### Phase 3: Secret Manager 統合

- [ ] Secret Manager API 有効化
- [ ] secrets 作成:
  - [ ] `dkmm-x-client-id`
  - [ ] `dkmm-x-client-secret`
  - [ ] `dkmm-x-refresh-token` (★rotate するからversion追加で運用)
  - [ ] `dkmm-claude-api-key`
  - [ ] `dkmm-slack-bot-token`
  - [ ] `dkmm-slack-signing-secret`
  - [ ] `dkmm-github-pat` (commit用)
- [ ] Cloud Run service account に IAM:
  - [ ] `secretmanager.secretAccessor`
  - [ ] `secretmanager.secretVersionAdder` (refresh_token 上書き用)
- [ ] `deploy.sh` を `--set-secrets` 形式に書き換え
- [ ] Python から `google-cloud-secret-manager` で読み書き実装

### Phase 4: bookmark fetcher を Cloud Run に移植

- [ ] `cloud_run/bookmark_fetcher.py` 作成 — 既存 `tools/fetch_bookmarks.py` ベース
  - [ ] CLIENT_ID/SECRET/REFRESH_TOKEN を Secret Manager から取得
  - [ ] 取得後の refresh_token を Secret Manager に書き戻し(version add)
  - [ ] state(`.seen_ids.json` 相当)も Firestore か GCS に永続化
- [ ] FastAPI に `POST /fetch` 追加 → fetcher 起動
- [ ] Cloud Scheduler 作成:
  ```
  gcloud scheduler jobs create http dkmm-bookmark-daily \
    --schedule="0 9 * * *" \
    --uri="https://.../fetch" \
    --http-method=POST \
    --oidc-service-account-email=...
  ```
- [ ] Cloud Run service の認証 OIDC 設定

### Phase 5: Claude API で draft 生成 + 軽量 index 維持

- [ ] `cloud_run/curator.py` 作成 — 新着ブックマークを Claude に渡して draft 生成
  - [ ] プロンプト設計: dkm 関心軸 + 著作権ルール(要約厚め, 丸写しNG) + ねーさんキャラ
  - [ ] 出力: thread draft (Markdown) + Slack 投稿用要約
- [ ] **★ 関連 thread 検索**:
  - [ ] `notes/_index.md` を最初に読む(なければ生成タスクを別途)
  - [ ] keyword/tag マッチで関連 thread 候補を抽出
  - [ ] full read せず Grep + 該当箇所のみ Read
- [ ] Slack channel に draft を投稿(slack-sdk)

### Phase 6: Slack 対話 → GitHub commit

- [ ] FastAPI の `/slack/events` で対話処理
  - [ ] dkm の Slack 反応を Claude に渡す
  - [ ] context は **関連 thread の該当箇所のみ** (★トークン効率)
- [ ] gh CLI または PyGithub で dkmm.com repo に commit/PR
- [ ] commit 時はファイルパスを `notes/{topic}/threads/YYYY-MM-DD_{slug}.md` 形式に従う(タイトル日本語)
- [ ] PR auto-merge or dkm 手動 merge を選択可能に

### Phase 7: index 自動メンテ

- [ ] thread の追加・更新を契機に `notes/_index.md` を再生成する Cloud Function or Cloud Run job
- [ ] embedding ベースに進化させる場合: Vertex AI Embeddings or text-embedding-3-small + Firestore

### Phase 8: 自分のツイート fetcher (オプショナル)

- [ ] `tools/fetch_my_tweets.py` (or `bookmark_fetcher.py` 一般化)
- [ ] dkm 自身の過去発言傾向を見直す用途

## 着手時の判断ポイント

- **どこから始める** → Phase 1 のスケルトン必須。動くものを Phase 単位で増築
- **bnv パターンとの差分管理** → Phase 1 ではほぼコピペ、Phase 3 で Secret Manager 統合の段階で乖離
- **テスト戦略** → Phase 1 直後に Slack 連携手動テスト、Phase 5 以降は Claude API モックで unit test
- **コスト監視** → Cloud Run は無料枠想定、Claude API は新着件数で変動。max budget 設定推奨
- **dry-run モード** → 実 Slack 投稿せず動作確認できるフラグを最初から仕込む

## 残ってる疑問

- Cloud Scheduler の認証は OIDC で Cloud Run 呼ぶのが標準だが、private endpoint の方が安全か?
- GitHub への commit は Personal Access Token か GitHub App か(個人 repo なら PAT で十分)
- Slack 対話の context window 管理。長い対話で trunc 発生したら要約に切り替える仕組み
- index ファイル方式 vs embedding 方式、どちらがコスト/精度のバランス良い?(まずは index ファイルから始めるべき)
- 失敗時の通知(Cloud Scheduler 失敗、Slack 投稿失敗)を Slack 別 channel に出す?

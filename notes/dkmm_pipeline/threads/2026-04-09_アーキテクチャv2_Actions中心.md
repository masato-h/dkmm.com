# アーキテクチャ v2: GitHub Actions + Claude Code 中心

> Cloud Run + Slack bot 構想を対話で再検討した結果、大幅にシンプル化。
> Cloud Run を捨てて GitHub Actions + Claude Code に一本化する方針に転換。

## きっかけ

- 前回セッション(2026-04-09)でロードマップ v1 を策定（Cloud Run 中心の 8 Phase 計画）
- Claude Max 契約の発覚でアーキテクチャ再検討が必要に
- 対話で要件を深掘った結果、以下が判明:
  - **通知フェーズはゆっくりでいい** → GitHub Actions で十分
  - **対話フェーズはさくさくやりたい** → Claude Code が最適（Slack bot では再現できない）
  - **Slack bot で対話を再現するにはコンテキスト管理を全部自前実装** → 割に合わない
  - **fetch は Claude Code から直接実行できる** → 自動化の優先度は「通知」だけ

## 引っかかったポイント

Slack bot で「今の Claude Code と同じ対話体験」を目指すと:
- API はステートレス → 会話履歴を Firestore 等に保存して毎回全部送り直し
- トークン上限管理、古い会話の要約圧縮、tool 呼び出し…全部自前実装
- Claude Code が裏でやってくれてることを再発明することになる
- **作る量に対して得られる体験の差が小さすぎる**

結論: Slack は通知に徹する。対話は Claude Code でやる。

## 全体像

```
自動（日次 cron）                 手動（気になった時）
──────────────                  ──────────────────
GitHub Actions                  dkm が Claude Code 開く
  ↓ fetch_bookmarks.py            ↓「Slack 読んで、○○深掘ろ」
  ↓ Claude Code Action             ↓ ねーさんが MCP で Slack 読む
  ↓ Slack に通知投稿                ↓ 対話 → thread 作成 → commit
```

★ 左半分が自動化スコープ。右半分は今と同じ。

## v1 からの変更点

| | v1 (Cloud Run 中心) | v2 (Actions + Claude Code) |
|---|---|---|
| fetch 実行環境 | Cloud Run job | GitHub Actions |
| draft 生成 | Cloud Run + Claude API (従量課金) | Claude Code Action (Max、無料) |
| Slack 通知 | Cloud Run → slack-sdk | GitHub Actions → Slack API |
| 対話 | Cloud Run Slack bot (自前 context 管理) | Claude Code (フレームワーク任せ) |
| OAuth token 永続化 | GCP Secret Manager | GCP Secret Manager (変更なし) |
| 状態管理 (.seen_ids) | Firestore / GCS | repo commit or Actions artifact |

### 捨てたもの
- Cloud Run (service + job 両方)
- Cloud Scheduler
- Firestore
- 自前の Slack bot コンテキスト管理コード
- Slack webhook 受信処理

### 残したもの
- GCP Secret Manager（X OAuth refresh_token の永続化に最適）
- Slack App（通知投稿用に必要、ただし bot 機能は不要）

## 設計上の制約（v1 から継続）

- ★スマホ完結: 通知は Slack で受動的に。対話したくなったら Claude Code を開くだけ
- ★ねーさんのトークン効率: 関連 thread 検索は index ベースで軽量に
- ★著作権ルール準拠: 要約は自分の言葉で5〜10行、原文丸写しNG
- ★コスト: Claude = Max で無料、GCP = Secret Manager のみでほぼゼロ

## 実装フェーズ

### Phase 0: 手動運用（★今ここ、もう動いてる）

dkm が Claude Code を開いて「ブックマーク取ってきて読んで」と言えば全部動く。

```
dkm「ブックマーク取ってきて」
  → ねーさんが python tools/fetch_bookmarks.py 実行
  → bookmarks/new.md (差分) を読む
  → 対話モードで深掘り
  → notes/{topic}/threads/ に thread 作成 → commit
```

これがベースライン。以降の Phase はこれを自動化・効率化していくだけ。

### Phase 1: GitHub Actions で bookmark 自動取得

手動の「取ってきて」を日次 cron に置き換える。

- [ ] `.github/workflows/fetch_bookmarks.yml` 作成
- [ ] `tools/fetch_bookmarks.py` を Actions 上で動くように調整
  - ファイルパスの解決（repo root 基準）
  - 依存パッケージ install（`pip install requests` 等）
- [ ] OAuth token 管理:
  - [ ] GCP Secret Manager に refresh_token を保存
  - [ ] GitHub Actions → Secret Manager アクセス（Workload Identity Federation）
  - [ ] 使用後の新 refresh_token を Secret Manager に書き戻し
- [ ] 差分管理 (.seen_ids.json):
  - [ ] repo に commit して永続化（artifact は 90 日で消える）
- [ ] cron: `0 0 * * *`（JST 9:00 = UTC 0:00）
- [ ] `workflow_dispatch` で手動トリガーも可能に
- [ ] 新着があれば `bookmarks/new.md` を commit して次の Phase へ

### Phase 2: Claude Code Action で draft 生成 + Slack 通知

新着検知 → ねーさんが要約 → Slack に投げる。

- [ ] `anthropics/claude-code-action` を GitHub Actions に導入
- [ ] Phase 1 の workflow から連鎖:
  - 新着あり → Claude Code Action に `bookmarks/new.md` を渡す
  - プロンプト設計:
    - dkm 関心軸を埋め込み
    - 著作権ルール（要約厚め、丸写しNG）
    - ねーさんキャラ
    - `notes/_index.md` 参照して既存 thread との関連示唆
  - 出力: Slack 投稿用の要約テキスト
- [ ] Slack 通知:
  - [ ] Slack App 作成（bot scope: `chat:write` のみ）
  - [ ] Bot token を GitHub Actions secrets に保存
  - [ ] `slackapi/slack-github-action` or Slack API 直叩きで投稿
  - [ ] 専用 channel: `#dkmm-bookmarks`
- [ ] dry-run モード（Slack 投稿せず Actions log に出力のみ）

### Phase 3: 対話フロー整備 + 軽量 index

Claude Code での対話効率を上げる。

- [ ] `notes/_index.md` 軽量カタログ作成
  - 全 thread のタイトル + keyword + 1行要約
  - セッション開始時にねーさんが読む → 関連 thread を即座に示唆
- [ ] 対話の定型フロー確立:
  1. dkm「Slack の新着読んで」
  2. ねーさんが MCP `slack_read_channel` で `#dkmm-bookmarks` を読む
  3. 気になるやつ選んで対話
  4. thread 作成 → `_index.md` 更新 → commit
- [ ] thread 作成時に `_index.md` を自動追記

### Phase 4: index メンテ + 関連発見（中期）

点と点を線で繋ぐ精度を上げる。

- [ ] `_index.md` の自動再生成（commit hook or 手動スクリプト）
- [ ] 新着ブックマークと既存 thread の keyword マッチで関連示唆を自動化
- [ ] 将来: embedding ベースの類似度検索（Vertex AI Embeddings + GCS）

### Phase 5: 自分のツイート fetcher（オプション）

- [ ] `tools/fetch_my_tweets.py`（or `fetch_bookmarks.py` を一般化）
- [ ] dkm の過去発言傾向を見直す用途

## 残ってる疑問

- **Workload Identity Federation vs サービスアカウントキー**: GitHub Actions → Secret Manager アクセス、どっちが楽？ WIF の方がセキュアだが初回セットアップが重い
- **.seen_ids.json の永続化**: repo commit だと毎日差分ノイズが出る。許容範囲か？
- **Claude Code Action の Max 利用上限**: 日次で叩いて上限に引っかからんか要確認
- **Slack App のスコープ**: 通知だけなら `chat:write` で十分。将来ボタン追加するなら `interactivity` も要る
